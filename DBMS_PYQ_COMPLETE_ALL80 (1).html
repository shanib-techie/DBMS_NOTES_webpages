2024 · Q1c Justify that Primary Key cannot be NULL 2 marks +

Entity Integrity Rule: No attribute of the primary key can take a NULL value. Justification:

  • NULL = unknown. If PK is NULL, we cannot uniquely identify that tuple — we don't know which entity it represents.
  • NULL ≠ NULL in SQL. Comparing NULL with any value (including another NULL) returns UNKNOWN, not TRUE. So uniqueness cannot be enforced for NULL PKs.
  • FK references break. Foreign keys in child tables reference the PK. A NULL PK means no meaningful FK reference is possible.
  • Uniqueness violated. Two tuples with NULL PKs cannot be distinguished — the relation degenerates.

2024 · Q1g Why is it necessary to give role names in a recursive relationship? 2 marks +

In a recursive (unary) relationship, the same entity type participates more than once in different semantic roles. Role names are necessary to:

  • Distinguish the two participations of the same entity type in the relationship.
  • Avoid ambiguity in the ER diagram and when mapping to relational tables.

Example: EMPLOYEE entity has a SUPERVISES relationship on itself.

  • Role 1: supervisor (the one who supervises)
  • Role 2: supervisee (the one being supervised)

Mapped table: SUPERVISES(supervisor_SSN, supervisee_SSN) — both FKs reference EMPLOYEE. Without role names, we cannot tell which SSN is which.

2023 · Q1i State the Lost Update problem in concurrent transaction processing 3 marks +

The Lost Update Problem occurs when two transactions T1 and T2 both read the same data item X, then both update X based on the original value. T2's write overwrites T1's write, so T1's update is "lost".

Example: X=50. T1 reads X=50, T2 reads X=50. T1 sets X=60 and writes. T2 sets X=55 (50+5) and writes, overwriting X=60. Final X=55 but should be 65.

Solution: Use locking — T2 must wait for T1 to release its write lock before T2 can read X. This ensures serializability.

2023 · Q2a Three-Schema Architecture — describe, diagram, why mappings needed ⭐ 9 marks 9 marks +

The three-schema architecture separates the database into three abstraction levels to achieve data independence:

LevelSchemaWhat it describesLanguage
ExternalExternal/View SchemaUser-specific views — each user/app sees only their relevant data portionVDL
ConceptualConceptual/Logical SchemaCommunity view of whole DB — entities, relationships, constraints, no physical detailsDDL
InternalInternal/Physical SchemaPhysical storage — file organization, indexes, access paths, record formatsSDL

Why mappings between levels?

  • External/Conceptual Mapping: Transforms between views and conceptual schema. Enables logical data independence — add new column to conceptual schema, update only this mapping, external schemas untouched.
  • Conceptual/Internal Mapping: Transforms between logical and physical. Enables physical data independence — change file organization, update only this mapping, conceptual schema untouched.
Without mappings, any change at one level forces changes at all levels — defeating the entire purpose of the architecture.

2018 · Q1a When is surrogate key necessary while mapping EER to relational DB? 2 marks +

A surrogate key is a system-generated unique ID (e.g., auto-increment integer) used as PK. It is necessary when:

  • The entity has no natural key (no attribute uniquely identifies it)
  • The natural key is too complex or large for efficient indexing and FK references
  • A weak entity is mapped — its identity depends on owner; surrogate replaces the partial key + owner PK combination
  • Natural keys can change over time (email, phone) — surrogate keys provide stable references

Example: ORDER_LINE weak entity → surrogate key OrderLineID instead of composite (OrderID, LineNo)

2018 · Q1d SQL clauses: ON UPDATE CASCADE and ON DELETE NULL — explain usage 2 marks +

ON UPDATE CASCADE: When the referenced PK value in the parent table is updated, the FK values in all child tables are automatically updated to match. Maintains referential integrity automatically.

ON DELETE SET NULL: When a referenced row in the parent table is deleted, the FK values in the child table are automatically set to NULL (instead of causing an error or cascading delete).

Example: Department table (parent) referenced by Employee (child via DeptNo FK):

  • ON UPDATE CASCADE: Change DeptNo 'D1' to 'D100' → all employees with DeptNo='D1' auto-update to 'D100'
  • ON DELETE SET NULL: Delete Department 'D1' → all employees in D1 get DeptNo=NULL
