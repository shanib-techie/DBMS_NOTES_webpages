ER Diagrams & Mapping to Relational Schema

Section A Q1(a) · Section B Q2(b) · Q5(b) · Q7(c)

🧱 What is an ER Diagram?

An Entity-Relationship (ER) diagram is a blueprint of a database. It shows what real-world things we want to store data about (entities), what they look like (attributes), and how they relate to each other (relationships).

🌍 Real-World Analogy

Think of a college database. Student, Course, and Professor are entities. A student enrolls in a course — that's a relationship. Student's name, roll number are attributes.

🔷 ER Diagram Symbols — Quick Cheatsheet

SymbolShapeMeaning

AttributeOval ○Property of entity (Name, RollNo)
Primary Key AttributeUnderlined OvalUniquely identifies entity
RelationshipDiamond ◇Association between entities
Composite AttributeOval with sub-ovalsMade of smaller parts (Name = First + Last)

🔗 Cardinality Ratios — How Many?

Cardinality tells you how many instances of one entity relate to how many instances of another.

1 : 1 (One-to-One) A person has exactly one passport. One passport belongs to one person.
1 : N (One-to-Many) One department has many employees. Each employee belongs to one department.
M : N (Many-to-Many) A student enrolls in many courses. A course has many students.
Ternary Relationship Involves 3 entities at once. E.g., Instructor guides Student on Project (ProjGuide from Q1a).

📌 Participation Constraints

Total Participation (double line =): Every entity MUST participate. E.g., every employee MUST work in a department.

Partial Participation (single line): An entity MAY participate. E.g., not every employee manages a department.

🗺️ Mapping ER Diagram → Relational Schema (Rules)

This is how we convert an ER diagram into actual tables:

  • 1

    Each entity becomes a table. Its attributes become columns. Primary key stays the same.

  • 2

    1:N relationship: Add the primary key of the "1-side" entity as a foreign key in the "N-side" table.

  • 3

    M:N relationship: Create a new table with primary keys of BOTH entities. These two together form the primary key of the new table.

  • 4

    Ternary relationship: Create a new table with primary keys of ALL THREE entities.

  • 5

    Multivalued attribute: Create a separate table with the entity's PK + the multivalued attribute.

🔍 SELECT with GROUP BY, HAVING, LIKE

-- GROUP BY: groups rows, aggregate per group SELECT deptName, COUNT(*), SUM(basicSalary) FROM retiredEmployee GROUP BY deptName; -- LIKE: pattern matching -- '_' = exactly 1 char, '%' = any chars SELECT empID, empName, deptName FROM retiredEmployee WHERE empName LIKE '_a%'; -- Matches: names where 2nd char is 'a' -- e.g., "Rahul" (R-a-h-u-l) ✅, "Naman" (N-a-m-a-n) ✅
💡 How '_a%' Works _ = skip first character. a = second character must be 'a'. % = anything after. So "Rahul" (2nd char = 'a'), "Naman" (2nd char = 'a') match. "Rohit" (2nd char = 'o') does NOT match.

❓ From Q1(c) — NATURAL JOIN vs Cartesian + WHERE

Given: Employee(empID, deptID, empSalary, empHobby) and Department(deptID, deptName, deptFloor)

-- Query (i): NATURAL JOIN — matches on deptID (same name in both) SELECT * FROM Employee E NATURAL JOIN Department D; -- Output columns: empID, deptID, empSalary, empHobby, deptName, deptFloor -- deptID appears ONCE (common column auto-merged) -- Query (ii): Cartesian product with WHERE filter SELECT * FROM Employee E, Department D WHERE E.deptID = D.deptID; -- Output columns: empID, E.deptID, empSalary, empHobby, D.deptID, deptName, deptFloor -- deptID appears TWICE (one from each table)
⚠️ Key Difference NATURAL JOIN eliminates duplicate columns. The WHERE-based join keeps all columns including duplicates. Same rows, different number of columns!

🔢 From Q1(e) — Theta Join on Tables R and S

Given R(A,B,C,D) and S(B,D,E). Theta join combines rows where condition is true.

📝 R ⋈(R.B=S.B) S — Match rows where R.B equals S.B

Go through each R row and find S rows where B values match:

R(15,1,15,a) matches S(1,a,15) → combined row: (15,1,15,a,1,a,15)

R(15,1,15,a) matches S(1,a,25) → combined row: (15,1,15,a,1,a,25)

R(20,2,25,a) matches S(2,b,30) → and so on...

📝 R ⋈(R.C=S.E) S — Match rows where R.C equals S.E

Compare each R.C with each S.E. S.E values are: 15, 20, 25, 30, null

R row with C=15 matches S row with E=15 → combine those rows

R row with C=25 matches S row with E=25 → combine those rows

R row with C=20 matches S row with E=20 → combine those rows

📝 From Q5(c) — Set Operations on T1 and T2

T1(P,Q,R) has rows: (10,a,5), (15,b,8), (25,a,6). T2(A,B,C) has rows: (10,b,6), (15,c,3), (10,b,5).

ρT (T1 × T2) — Rename + Cartesian Product

T1 × T2 = every T1 row combined with every T2 row → 3×3 = 9 rows. ρT just renames the result table to T.

T1 − T2 (Difference)

For set operations, tables must be union-compatible (same number of columns, compatible types). T1 has 3 cols, T2 has 3 cols ✅. T1 − T2 = rows in T1 that are NOT in T2 (comparing corresponding values). Since (10,a,5) ≠ (10,b,6) etc., all T1 rows are kept: result = {(10,a,5),(15,b,8),(25,a,6)}.

T1 ∩ T2 (Intersection)

Rows that exist in BOTH T1 and T2 (exact match). No row in T1 exactly matches any row in T2, so result = ∅ (empty).

Functional Dependencies & Closure

Section A Q1(j) · Section B Q3(c) · Q4(a) · Q6(d)

📐 What is a Functional Dependency?

A functional dependency X → Y means: if two rows have the same value of X, they MUST have the same value of Y. X "determines" Y.

🌍 Simple Example

In a Student table: RollNo → Name

If two rows have the same RollNo, they MUST have the same Name. One roll number can't belong to two different names.

But Name → RollNo is NOT a FD (two students can have the same name).

📏 Armstrong's Axioms (Inference Rules)

RuleNameMeaning
If Y ⊆ X, then X → YReflexivityA set always determines its own subsets. {A,B} → A
If X → Y, then XZ → YZAugmentationAdd same attribute to both sides
If X→Y and Y→Z, then X→ZTransitivityChain dependencies together
If X→Y and X→Z, then X→YZUnionCombine right-hand sides
If X→YZ, then X→Y and X→ZDecompositionSplit right-hand side
If XY→Z and X→Y, then X→ZPseudotransitivityReplace Y with what X determines
📝 From Q3(c) — Pseudotransitivity Rule

Rule: If X → Y and WY → Z, then WX → Z

Proof: Given X → Y. Augment both sides with W: WX → WY. We know WY → Z. By transitivity: WX → Z. ✅

Apply to F = {M → P, MY → P, YP → C}:

We have M → P. Augment: MY → PY → ... Use YP → C and M → P: MP → C... Actually: from M → P, augment with Y: MY → YP. We know YP → C. By transitivity: MY → C. This is the new FD inferred! ✅

🔍 Attribute Closure — Finding {X}+

The closure of a set X (written X⁺) is all attributes that X can determine using all functional dependencies.

  • 1

    Start with X⁺ = X (the set itself)

  • 2

    Look through all FDs. If the left side is a subset of X⁺, add the right side to X⁺

  • 3

    Repeat until no new attributes can be added

📝 From Q1(j) — Find {A,B}⁺ with F = {{A,B}→C, {A}→{D,E}, {B}→F, {F}→{G,H}, {D}→{I,J}}

Start: {A,B}⁺ = {A, B}

Apply {A,B} → C: {A,B}⁺ = {A, B, C}

Apply {A} → {D,E}: {A,B}⁺ = {A, B, C, D, E}

Apply {B} → F: {A,B}⁺ = {A, B, C, D, E, F}

Apply {F} → {G,H}: {A,B}⁺ = {A, B, C, D, E, F, G, H}

Apply {D} → {I,J}: {A,B}⁺ = {A, B, C, D, E, F, G, H, I, J}

Final: {A,B}⁺ = {A,B,C,D,E,F,G,H,I,J} — ALL attributes in R! So {A,B} is a candidate key.

🔍 Checking FDs from a Table Instance — Q6(d)

Given a state of R(A,B,C,D): rows (1,2,3,4),(1,2,3,5),(6,7,8,2),(2,1,3,4). Check which FDs MAY hold.

🧠 Key Rule An FD X → Y CANNOT hold if there exist two rows with the SAME X values but DIFFERENT Y values.
📝 Checking each FD

(i) D → A: D=4 appears twice (rows 1 & 4) with A=1 and A=2 (different!). ❌ CANNOT hold.

(ii) BC → D: BC=(3,4) appears twice (rows 1 & 2) with D=4 and D=5 (different!). ❌ CANNOT hold.

(iii) BC → A: BC=(3,4) appears twice with A=1 and A=1 (same! rows 1&2). BC=(8,2) once. BC=(3,4) once again... All same BC give same A. ✅ MAY hold.

Normalization — 1NF, 2NF, 3NF, BCNF

Section B Q2(a) · Q2(c) · Q3(c) · Q4(a) · Q6(c)

💡 Why Normalize?

Normalization removes data redundancy (repeated data) and anomalies from database tables.

❌ Update Anomaly If "C++ course credits" is stored many times, updating it in one row but not others → inconsistency.
❌ Insertion Anomaly Can't add a course without at least one student enrolled.
📝 From Q1(i) — StudentCourse update anomaly

Table: StudentCourse(studentID, studentName, CGPA, courseID, courseName, credits)

DELETE studentID=132: Shyam is the only one in Java (C121). Deleting him loses the Java course info → Deletion anomaly (not update anomaly).

UPDATE credits=3 WHERE courseID=118: C++ (C118) is taken by students 123, 131, 135. Changing credits must be done in ALL 3 rows. If you miss one → Update anomaly ✅

📊 Normal Forms — Step by Step Ladder

1NF — First Normal Form Rule: Every cell must have a single atomic value. No repeating groups or arrays in a cell.
Example: Phone = "9999, 8888" in one cell → NOT 1NF. Split into separate rows → 1NF ✅
2NF — Second Normal Form Rule: Must be in 1NF + No partial dependency (every non-key attribute must depend on the WHOLE primary key, not just part of it).
Only relevant when PK is composite!
Partial Dependency: Non-key attribute depends on PART of composite PK.
3NF — Third Normal Form Rule: Must be in 2NF + No transitive dependency (non-key attributes must NOT depend on other non-key attributes).
Transitive Dependency: A → B → C where A is PK, B and C are non-key.
BCNF — Boyce-Codd Normal Form Rule: For every FD X → Y, X must be a superkey. Stricter than 3NF.
A relation with only 2 attributes is always in BCNF!

🔧 Normalization Example — CarSale (From Q2(c))

CarSale(Car#, dateSold, Salesperson#, Commission%, DiscountAmt)
PK = {Car#, Salesperson#}
Extra FDs: dateSold → DiscountAmt, Salesperson# → Commission%

Step 1: Check 2NF — Look for partial dependencies

PK = {Car#, Salesperson#}. Non-key attrs: dateSold, Commission%, DiscountAmt.

dateSold → DiscountAmt: dateSold is NOT part of PK. But DiscountAmt depends only on dateSold, not full PK... Wait — dateSold is a non-key! DiscountAmt depends on a non-key (transitive, not partial).

Salesperson# → Commission%: Salesperson# IS part of PK. Commission% depends only on Salesperson#, not full PK {Car#, Salesperson#}. → PARTIAL DEPENDENCY! NOT in 2NF.

Step 2: Convert to 2NF — Remove partial dependencies

R1: CarSale(Car#, dateSold, Salesperson#) — non-key attrs fully dependent on whole PK

R2: Salesperson(Salesperson#, Commission%) — partial dep extracted

Now check for transitive deps in R1: dateSold → DiscountAmt. DiscountAmt is a transitive dep through dateSold.

Step 3: Convert to 3NF — Remove transitive dependencies

From R1, split out: SaleDiscount(dateSold, DiscountAmt)

Final 3NF Tables:

🔹 CarSale(Car#, dateSold, Salesperson#) — PK: {Car#, Salesperson#}

🔹 Salesperson(Salesperson#, Commission%) — PK: Salesperson#

🔹 SaleDiscount(dateSold, DiscountAmt) — PK: dateSold

🔍 BCNF Check — From Q2(a)

Relation R(A, B) — Only two attributes. PK = A (or B if it's the PK — underlined A means A is PK).

BCNF Rule Check For every FD X → Y, X must be a superkey. In R(A,B) with PK=A:
• A → B: A is PK = superkey ✅
• B → A: only if B is also a key (if B uniquely identifies rows too).
A 2-attribute relation is ALWAYS in BCNF unless there are anomalous FDs beyond the key. Generally: R(A,B) with PK=A is in BCNF if no FD exists where LHS is a non-superkey.

🔗 Lossless Join Decomposition — From Q6(c)

When we split a relation R into R1 and R2, the decomposition is lossless if we can get the original R back by joining R1 and R2 (no spurious tuples).

Lossless Condition: R1 ∩ R2 → R1 OR R1 ∩ R2 → R2
📝 R(A,B,C) → R1(A,B) and R2(B,C)

R1 ∩ R2 = {B} (common attribute)

Lossless if: B → {A,B} i.e., B → A OR B → {B,C} i.e., B → C

So the decomposition is lossless if B is a key of R1 (B → A) or B is a key of R2 (B → C).

Transactions & Concurrency Problems

Section A Q1(h) · Q5(c)

📦 What is a Transaction?

A transaction is a unit of work — a sequence of operations (read, write) that should execute completely or not at all. Think of it like a bank transfer: either both debit and credit happen, or neither does.

ACID Properties Atomicity: All or nothing. Consistency: DB stays valid before & after. Isolation: Transactions don't interfere. Durability: Committed changes are permanent.

⚠️ Concurrency Problems — From Q1(h)

The schedule: T1 reads X, T1 sets X=X+10, T2 reads X, T2 sets X=X+20, T1 writes X, T2 writes X. Initial X=5.

TimeT1T2X in memory
1read(X) → X=55
2X = X+10 = 1515 (local)
3Read(X) → X=55 (reads original!)
4X = X+20 = 2525 (local)
5Write(X) = 1515
6Write(X) = 2525
7Commit25
8Commit25
❌ Problem: Lost Update T1's update (X=15) was overwritten by T2's write (X=25). T1's +10 is completely LOST.
Correct value should be: 5+10+20 = 35, but we got 25!
This is called the Lost Update Problem — a classic concurrency issue.

3-Schema Architecture & Data Independence

Section B Q3(b) · Q6(a)

🏗️ Three Levels of Database Architecture

ANSI/SPARC 3-Schema architecture separates how data is stored from how it is viewed.

  • 1

    External Level (View Level): What individual users or applications see. Different users see different views. Example: HR sees employee salaries; other employees only see names and departments.

  • 2

    Conceptual Level (Logical Level): The complete logical structure of the whole database — all tables, relationships, constraints. DBA works at this level. Example: The full schema with Student, Course, Enroll tables and all FKs.

  • 3

    Internal Level (Physical Level): How data is actually stored on disk — file formats, indexes, storage structures. Example: B+ tree indexes on RollNo, heap files for data storage.

    • 🔄 Data Independence

    Logical Data Independence Change the conceptual schema WITHOUT affecting external views/apps. Example: Add a new column "email" to Student table — existing apps that don't use email still work fine.
    Physical Data Independence Change the internal storage WITHOUT affecting the conceptual schema. Example: Change from heap file to B+ tree — your SQL queries don't need to change.

    📁 File Processing vs Database Approach — From Q6(a)

    AspectFile ProcessingDatabase Approach
    Self-describing❌ Structure defined in application code✅ Metadata (catalog) stored in DB itself
    Data sharing❌ Each app has its own files✅ Multiple users share same DB
    Redundancy❌ Same data stored in multiple files✅ Centralized, minimal redundancy
    Inconsistency❌ Updates in one file miss others✅ Single source of truth
    Data independence❌ Tightly coupled✅ Physical & logical independence
    💡 Self-Describing Nature A DB is self-describing because it stores its own structure (schema, table definitions, data types) in a system catalog / data dictionary — separate from the actual data. File systems don't have this.

    Indexing & File Organization

    Section B Q6(b) · Q7(a) · Q7(b)

    📚 What is an Index?

    An index is like the index at the back of a textbook. Instead of reading every page to find "binary search", you look it up in the index which tells you exactly which page. Similarly, a DB index tells you exactly which disk block a record is in.

    🗂️ Types of Indexes

    TypeBased onKey Property
    Primary IndexOrdering key field of ordered fileOne per file, data physically ordered by this field
    Clustering IndexNon-key ordering fieldRecords grouped by this field value
    Secondary IndexNon-ordering fieldMultiple allowed; data NOT ordered by this field
    Dense IndexOne entry per recordFaster search, more space
    Sparse IndexOne entry per blockLess space, file must be ordered
    💡 From Q6(b) — Why only ONE primary index but MULTIPLE secondary indexes? A primary index is based on the physical ordering of data on disk. A file can only be physically ordered in ONE way, so only ONE primary index is possible. Secondary indexes are separate structures that point to records by different fields — they don't control physical order, so you can have as many as you want (on Name, on Department, on Salary, etc.).

    🔢 Multilevel Indexing — From Q7(a)

    If the index itself is too large to fit in memory, we create an index of the index. This is multilevel indexing.

  • 1

    Level 1 (Inner Index): Points to data records. Could be 100s of blocks.

  • 2

    Level 2 (Outer Index): Index over the level-1 index. Much smaller.

  • 3

    Search: Binary search top-level → find level-1 range → find data block. Much fewer disk accesses.

    • 💡 Why it's faster Instead of binary searching 1000 index blocks (log₂1000 ≈ 10 accesses), with multilevel indexing you might only need 2-3 disk accesses. B+ Trees are the most common implementation of multilevel indexing.

    🧮 File Storage Calculations — From Q7(b)

    Given: r=10,000 records, V=9 bytes (search key), remaining=91 bytes, B=1024 bytes (block size), unspanned, fixed-size records.

    Record Length (R) = V + remaining = 9 + 91 = 100 bytes
    Blocking Factor (bfr) = ⌊ B / R ⌋ = ⌊ 1024 / 100 ⌋ = 10 records per block
    Number of File Blocks (b) = ⌈ r / bfr ⌉ = ⌈ 10,000 / 10 ⌉ = 1,000 blocks
    Binary Search Block Accesses = ⌈ log₂(b) ⌉ = ⌈ log₂(1000) ⌉ = ⌈ 9.97 ⌉ = 10 block accesses
    📝 How Binary Search Works on Disk 1. Access middle block. 2. If key found → done. 3. If search key smaller → go left half. 4. If larger → go right half. 5. Repeat. With 1000 blocks, needs ⌈log₂(1000)⌉ = 10 disk accesses maximum.

    Quick Revision — Formulas & Rules

    Last-Minute Cheatsheet

    🔑 Normal Forms Summary

    1NF: Atomic values, no repeating groups

    2NF: 1NF + No partial dependency (composite PK)

    3NF: 2NF + No transitive dependency

    BCNF: Every determinant is a superkey

    🧮 File Storage Formulas

    R = V + rest (record length)

    bfr = ⌊B/R⌋ (blocking factor)

    b = ⌈r/bfr⌉ (file blocks)

    Binary search = ⌈log₂(b)⌉ accesses

    📐 Key FD Rules

    Reflexivity: X → subset of X

    Augmentation: X→Y ⟹ XZ→YZ

    Transitivity: X→Y, Y→Z ⟹ X→Z

    Pseudotransitivity: X→Y, WY→Z ⟹ WX→Z

    🔗 ER Mapping Rules

    Entity → Table

    1:N → FK on N-side

    M:N → New table with both PKs

    Ternary → New table with all 3 PKs

    Multivalued → Separate table
