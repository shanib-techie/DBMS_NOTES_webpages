SECTION 01

What is Three-Schema Architecture?

Imagine a hospital. The doctor sees a patient's health summary. The administrator sees billing records. The server in the basement stores thousands of files. Everyone sees the same data, but in different ways. That's exactly what Three-Schema Architecture does for databases.

The Big Idea

Three-Schema Architecture (also called ANSI/SPARC architecture, proposed in 1975) separates a database into 3 distinct levels so that changes in one level don't break the others. This is how databases achieve data independence.

👁 EXTERNAL LEVEL
What different users SEE
🧩 CONCEPTUAL LEVEL
What the DATABASE contains
💾 INTERNAL LEVEL
How data is PHYSICALLY stored

SECTION 02

Deep Dive: The 3 Layers

Let's understand each layer with a real bank database example.

📌 REAL EXAMPLE: Bank Database

External Level — User View

Customer logs in and sees only their own account balance, transactions, and name. A bank manager sees all accounts for their branch. A teller sees different fields.

VIEW customer_view AS
SELECT name, balance
FROM accounts
WHERE owner = current_user;
Conceptual Level — Full DB Map

The DBA sees the complete database design: tables for Accounts, Customers, Loans, Transactions, Branches — all columns, data types, and relationships between them.

TABLE Accounts (
  acc_id INT PRIMARY KEY,
  owner VARCHAR(50),
  balance DECIMAL(10,2),
  branch_id INT
);
Internal Level — Physical Storage

The DBMS engine stores data in B-tree indexes, allocates disk blocks, compresses records, and organizes files on SSD/HDD. Users never see this layer.

/* Stored as: */
B-tree index on acc_id
Hash index on owner
Clustered on branch_id
Compressed blocks: 4KB

WHY THREE LEVELS?

Without this separation, every time a DBA changes how data is stored on disk (internal), every app would break. Every time a new table is added (conceptual), every user view would need rewriting. The three levels create walls between concerns, so each can change independently.

SECTION 03 — QUESTION ANSWER

Schema Definition Languages

A Schema Definition Language (SDL) is a language used to define (describe/create) the structure of a database at a particular level of the three-schema architecture. Think of it as the "language used to write the blueprint" for each layer.

UNDERSTAND THE QUESTION FIRST

The question asks: "Name 3 languages that are used to define schemas — and tell us which level (external / conceptual / internal) each one supports."

Simple version: Which languages write the rules for each of the 3 floors of our building?

01

DDL

Data Definition Language

DDL is the part of SQL used to define the conceptual and internal schema. When you write CREATE TABLE, you're writing DDL. You're telling the database what entities exist, what columns they have, and what types they are.

Some advanced DDL commands also let you define storage parameters (how data is physically stored), touching the internal level too.

CREATE TABLE Student (
  student_id INT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(100) UNIQUE,
  dob DATE
);

/* Internal-level DDL example */
CREATE INDEX idx_name
ON Student(name);
Conceptual Level
Internal Level (partly)
02

SQL (VIEW / VDL)

View Definition Language — SQL CREATE VIEW

The CREATE VIEW statement in SQL acts as the View Definition Language (VDL) — it defines the external schema. Each view is a "window" into the database that a specific user or application sees.

Different users get different views — a student sees their own grades, a professor sees their class list, an admin sees everything. None of them see the raw tables directly.

/* External Schema — for students */
CREATE VIEW student_portal AS
SELECT s.name, c.course_name, g.grade
FROM Student s
JOIN Enrollment e ON s.id = e.student_id
JOIN Course c ON e.course_id = c.id
JOIN Grades g ON e.id = g.enroll_id;

/* Professor sees different view */
CREATE VIEW prof_class_list AS
SELECT s.name, s.roll_no...
External Level ✅ Primary
03

SDL

Storage Definition Language (Internal SDL)

SDL (Storage Definition Language) defines the internal schema — the physical structure of stored data. It specifies how data is actually saved on disk: file formats, record lengths, indexes, and access paths.

This is used mostly by the DBMS internally. DBAs may write SDL when fine-tuning performance. Modern systems embed SDL features inside DDL commands with storage clauses.

/* SDL / Internal Schema definition */
STORED RECORD TYPE Student
  LENGTH 256 bytes
  FIELD student_id: INTEGER(4)
  FIELD name: CHAR(100)
  ACCESS PATH: B-TREE
  ON student_id;

/* Modern equivalent in MySQL */
CREATE TABLE Student (...)
ENGINE=InnoDB ROW_FORMAT=COMPRESSED
KEY_BLOCK_SIZE=4;
Internal Level ✅ Primary
SECTION 04

Mappings Between Levels

The three levels don't just exist alone — the DBMS connects them using mappings. A mapping translates requests from one level to another.

👁 External Schema

User queries a view:
"Give me student grades"

⇄
E/C MAPPING

🧩 Conceptual Schema

DBMS translates view query into full table query across Student, Enrollment, Grades tables

🧩 Conceptual Schema

DBMS has a logical query:
"Get rows from Student table"

⇄
C/I MAPPING

💾 Internal Schema

DBMS translates to physical: read B-tree index, fetch disk block #4821, decompress 34 bytes

DATA INDEPENDENCE COMES FROM MAPPINGS

Because mappings translate between levels, you can change the internal level (e.g., switch from B-tree to hash index) and only update the C/I mapping — the conceptual and external levels remain unchanged. That's physical data independence. Similarly, adding a new table only changes the E/C mapping — existing views still work. That's logical data independence.

EXAM ANSWER — WRITE THIS

1. DDL (Data Definition Language) — supports the Conceptual Level. Used to define tables, columns, keys, and constraints. Also partially supports the Internal level via CREATE INDEX and storage parameters.

2. VDL / SQL CREATE VIEW — supports the External Level. Used to define user-specific views, hiding underlying table complexity.

3. SDL (Storage Definition Language) — supports the Internal Level. Used to define how data is physically stored on disk — record formats, indexes, file organizations.
