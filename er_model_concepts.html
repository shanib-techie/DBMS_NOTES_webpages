Section 01

What is the ER Model?

The Entity-Relationship (ER) Model is a high-level conceptual data model used to design a database at the logical level — before any SQL or tables are created. Proposed by Peter Chen in 1976, it lets designers visualize real-world objects (entities), their properties (attributes), and how they relate to one another (relationships).

Component 01

Entity

A real-world object or "thing" distinguishable from others. Example: MOVIE, ACTOR, DIRECTOR. Represented as a rectangle in ER diagrams.

Component 02

Attribute

A property or characteristic of an entity or relationship. Example: name, date_of_birth, title. Represented as an ellipse/oval.

Component 03

Relationship

An association between two or more entities. Example: ACTOR "appears in" MOVIE. Represented as a diamond.

Component 04

Entity Set

A collection of similar entities. All movies form the MOVIE entity set. Each entity in the set shares the same attributes.

Component 05

Key Attribute

An attribute (or set of attributes) that uniquely identifies each entity in an entity set. Shown with an underline. Example: title + year for MOVIE.

Component 06

Cardinality

Defines how many instances of one entity relate to another: 1:1, 1:N, or M:N. Written on relationship lines.

Concept 07

Weak Entity

An entity that cannot be uniquely identified by its own attributes alone — it depends on a strong/owner entity. Drawn with a double rectangle. Its identifying relationship uses a double diamond.

Concept 08

Participation Constraint

Total participation (double line) — every entity must participate in the relationship. Partial participation (single line) — some entities may not participate.

Concept 09

Relationship Degree

Unary (1 entity), Binary (2 entities — most common), Ternary (3 entities). Example: ACTOR–MOVIE–ROLE is ternary.

Section 02

Types of Attributes

Attributes describe entity properties. There are six main types — each has a different ER notation.

Type Symbol / Notation Description Example
Simple (Atomic) ○ Plain oval Cannot be divided further. Most basic type. Age, Gender, Length
Composite ○ Oval with child ovals Can be split into smaller sub-parts, each of which is meaningful. Name → {First, Last}
Address → {Street, City, ZIP}
Single-valued ○ Plain oval Holds exactly ONE value per entity instance. Date of Birth, Movie Title
Multi-valued ◎ Double oval Can hold MULTIPLE values for one entity instance. Genre (Horror, Action, Drama…)
Phone Numbers
Derived ○ Dashed oval Value is calculated/derived from another attribute — not stored directly. Age (derived from Date of Birth)
Duration in hours (from minutes)
Key Attribute ○ Underlined name Uniquely identifies each entity. Can be simple or composite. StudentID, Title+Year
Null Attribute ○ Plain oval An attribute that may have no applicable value for some entities (optional). Middle Name, Plot Outline

Section 04 — Q2 Answer

Data Independence

Data independence is the ability to modify the schema at one level of the database architecture without affecting the schema at the next higher level. It is one of the main goals of the three-level ANSI/SPARC architecture (External → Conceptual → Internal).

There are two types:

1

Physical Data Independence

The ability to change the internal/physical schema (how data is stored on disk — file organization, indexing, compression) without changing the conceptual schema or any application programs.

Example: Switching storage from B-tree index to hash index, or moving data from HDD to SSD, without altering the logical table structure or any SQL queries.
2

Logical Data Independence

The ability to change the conceptual/logical schema (adding/removing tables, attributes, or relationships) without changing the external schema (user views) or application programs.

Example: Adding a new column "budget" to the MOVIE table without breaking existing queries or views that don't use that column.
Why it matters: Logical data independence is harder to achieve than physical because changes to the logical structure often ripple through application logic and views. DBMSs use view mechanisms and mapping between levels to enforce both types of independence.

Bonus: University Database ER Diagram

This is a self-created example — a University Database tracking students, courses, instructors, and departments.

Requirements summary: Each Student (identified by student_id) enrolls in one or more Courses. Each Course has a course_id, name, and credits, and belongs to a Department. Each Instructor (identified by instructor_id) teaches one or more courses and belongs to a department. A department has a name and building. Enrollment has a grade attribute.
