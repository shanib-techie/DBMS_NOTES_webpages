01

What is a JOIN? — The Big Idea

Simple Definition

A JOIN is used to combine rows from two or more tables based on a related column between them.

In real databases, data is split across many tables (to avoid repetition). JOIN brings them back together when you need to see combined info.

🌍 Real Life Analogy

Think of two lists:

  • List 1 (Students): Student ID, Student Name
  • List 2 (Marks): Student ID, Subject, Marks

If you want to see "Rahul scored 85 in Maths" — you need to JOIN both lists using Student ID as the common link. That's exactly what SQL JOIN does!

Why not put everything in one table? If we stored student name + marks in one table, we'd repeat the student name for every subject. That's data redundancy. So we use JOIN to combine when needed.
General SQL JOIN Syntax
SELECT columns
FROM   Table1
JOIN_TYPE Table2
ON     Table1.common_column = Table2.common_column;

Replace JOIN_TYPE with: INNER JOIN, LEFT JOIN, RIGHT JOIN, etc.

03

INNER JOIN — Only Matched Rows

INNER JOIN Returns rows that have matching values in BOTH tables
🌍 Real Life Example

You have a guest list (invited people) and a check-in list (people who actually came). INNER JOIN = only people who are on BOTH lists (invited AND came). If someone was invited but didn't come, they're OUT. If someone came but wasn't on the list, they're also OUT.

SQL Query
SELECT s.student_id, s.student_name, e.course, e.marks
FROM   STUDENTS s
INNER JOIN ENROLLMENTS e
ON     s.student_id = e.student_id;
Result
student_idstudent_namecoursemarks
1RahulMaths85
1RahulEnglish78
2PriyaScience90
3AmitMaths72
❌ Missing from resultSara (ID=4) is NOT here — she has no enrollment. student_id=99 is NOT here — no matching student. INNER JOIN only keeps perfect matches!
💡 Memory Trick
INNER JOIN = Intersection = Only the common part. Like a Venn diagram — only the overlapping middle section.
04

LEFT JOIN — All Left + Matched Right

LEFT JOIN Returns ALL rows from LEFT table + matched rows from RIGHT table
🌍 Real Life Example

You're a teacher taking attendance. You have a class roll list (all students) and a homework submission list. LEFT JOIN = show ALL students, and if they submitted homework, show that too. If a student didn't submit — still show them, but put NULL in the homework column.

SQL Query
SELECT s.student_id, s.student_name, e.course, e.marks
FROM   STUDENTS s
LEFT JOIN ENROLLMENTS e
ON     s.student_id = e.student_id;
Result
student_idstudent_namecoursemarks
1RahulMaths85
1RahulEnglish78
2PriyaScience90
3AmitMaths72
4SaraNULLNULL
✅ Sara is included now!Sara (ID=4) appears with NULL in course and marks — she's a student, but hasn't enrolled. LEFT JOIN never drops anyone from the LEFT table.
Practical Use Case — Find students with NO enrollment
-- Find students who haven't enrolled in any course
SELECT s.student_name
FROM   STUDENTS s
LEFT JOIN ENROLLMENTS e ON s.student_id = e.student_id
WHERE  e.student_id IS NULL;

-- Result: Sara
💡 Memory Trick
LEFT JOIN = Left table is the BOSS. All rows from left, no matter what. Right table fills in where it can, puts NULL where it can't.
05

RIGHT JOIN — All Right + Matched Left

RIGHT JOIN Returns ALL rows from RIGHT table + matched rows from LEFT table
🌍 Real Life Example

Imagine you have a list of all courses offered and a list of enrolled students. RIGHT JOIN = show ALL enrollment records, and if the student exists in student table, show their name. If the enrollment has a student_id that doesn't exist — still show enrollment, put NULL for student details.

SQL Query
SELECT s.student_id, s.student_name, e.course, e.marks
FROM   STUDENTS s
RIGHT JOIN ENROLLMENTS e
ON     s.student_id = e.student_id;
Result
student_id (from S)student_namecoursemarks
1RahulMaths85
2PriyaScience90
1RahulEnglish78
3AmitMaths72
NULLNULLHistory65
⚠ The orphan enrollment appears!The enrollment with student_id=99 (History, 65 marks) appears — but since no student with ID=99 exists, the student columns show NULL. Sara is NOT here because she has no enrollment row.
💡 Memory Trick
RIGHT JOIN = Exact OPPOSITE of LEFT JOIN. Right table is the boss now. Simply swap the table order + use LEFT JOIN for the same result.
💡 Pro TipMost developers avoid RIGHT JOIN — instead, they just swap the table order and use LEFT JOIN. Both give same result. LEFT JOIN is more commonly used.
06

FULL OUTER JOIN — Everything from Both Tables

FULL OUTER JOIN Returns ALL rows from BOTH tables — matched and unmatched
🌍 Real Life Example

You have two companies merging. Company A's employee list + Company B's employee list. FULL OUTER JOIN = show EVERYONE from both companies. If someone works in both — show once with combined data. If only in Company A — show with NULL for Company B fields. If only in Company B — show with NULL for Company A fields.

SQL Query
SELECT s.student_id, s.student_name, e.course, e.marks
FROM   STUDENTS s
FULL OUTER JOIN ENROLLMENTS e
ON     s.student_id = e.student_id;
Result
student_idstudent_namecoursemarksNote
1RahulMaths85Matched
1RahulEnglish78Matched
2PriyaScience90Matched
3AmitMaths72Matched
4SaraNULLNULLLeft only
NULLNULLHistory65Right only
Result ExplanationMatched rows appear once. Sara (only in STUDENTS) appears with NULL enrollment. The ID=99 enrollment (only in ENROLLMENTS) appears with NULL student info. FULL OUTER JOIN = LEFT JOIN + RIGHT JOIN combined (minus duplicates).
⚠ MySQL NoteMySQL does NOT support FULL OUTER JOIN directly. Workaround: Use LEFT JOIN UNION RIGHT JOIN.
-- MySQL workaround for FULL OUTER JOIN
SELECT s.student_name, e.course FROM STUDENTS s LEFT JOIN ENROLLMENTS e ON s.student_id = e.student_id
UNION
SELECT s.student_name, e.course FROM STUDENTS s RIGHT JOIN ENROLLMENTS e ON s.student_id = e.student_id;
💡 Memory Trick
FULL OUTER JOIN = LEFT JOIN + RIGHT JOIN = "Nobody gets left behind." Every row from everywhere shows up.

CROSS JOIN — Every Row × Every Row

CROSS JOIN Cartesian Product — Every row from Table1 combined with every row from Table2
🌍 Real Life Example

You run a T-shirt shop. You have 3 colors (Red, Blue, Green) and 3 sizes (S, M, L). How many combinations? 3 × 3 = 9. CROSS JOIN generates ALL possible combinations — Red-S, Red-M, Red-L, Blue-S, Blue-M ... That's a Cartesian product.

SELF JOIN — A Table Joining Itself

SELF JOIN A table is joined with itself using different aliases
🌍 Real Life Example

You have an EMPLOYEE table where each employee has a manager_id. The manager is ALSO an employee in the same table! To find "who manages whom" — you need to JOIN the EMPLOYEE table with itself. One copy = employees, other copy = managers.

Table: EMPLOYEES

SQL Query — Find Employee + Their Manager Name
SELECT

FROM   EMPLOYEES e
INNER JOIN EMPLOYEES m   -- Same table, different alias

💡 Memory Trick
SELF JOIN = Table talks to itself using two different nicknames (aliases). Always use aliases: e for employee, m for manager.
09

NATURAL JOIN — Automatic Column Matching

NATURAL JOIN Automatically joins on ALL columns with the SAME NAME in both tables
🌍 Real Life Example

Imagine two spreadsheets that both have a column called "student_id". NATURAL JOIN says: "I'll figure out which column to join on by myself — any column with the same name in both sheets." No need to write ON condition. It automatically finds student_id and joins on it.

SELECT *
FROM   STUDENTS
NATURAL JOIN ENROLLMENTS;

-- Automatically joins on: student_id (same name in both tables)
-- No ON clause needed!
✅ Advantage Shorter, cleaner syntax. No need to write ON clause. Good when tables are well-designed with consistent column names.
❌ Danger If both tables have multiple same-named columns (like "name"), it joins on ALL of them — which can give wrong/unexpected results!
💡 Memory Trick
NATURAL JOIN = Lazy JOIN. It finds common column names on its own. But be careful — too "lazy" can cause bugs if you're not careful about column names.
10

EQUI JOIN vs NON-EQUI JOIN

EQUI JOIN Join using = (equals)
🌍 Example

Match students where student_id is equal in both tables. This is what we've been doing all along — the standard JOIN using =.

SELECT * FROM STUDENTS s
JOIN ENROLLMENTS e
ON s.student_id = e.student_id;

Uses = operator — most common type of JOIN.

NON-EQUI JOIN Join using <, >, BETWEEN, !=
🌍 Real Life Example

A bank has a salary table and a tax bracket table. To find which tax bracket each employee falls into — salary is BETWEEN a range, not equal to something.

-- Salary vs Tax Bracket
SELECT e.name, e.salary, t.tax_rate
FROM   EMPLOYEES e
JOIN   TAX_BRACKETS t
ON     e.salary BETWEEN t.min_sal AND t.max_sal;

Uses BETWEEN — not equality.

TypeOperator UsedCommon Use Case
EQUI JOIN=Match IDs, match foreign keys (most common)
NON-EQUI JOIN<, >, <=, >=, BETWEEN, !=Salary ranges, date ranges, score grades

JOIN vs WHERE Condition vs Subquery

Same Result — 3 Different Ways

Task: "Get names of students who enrolled in Maths"

Method 1 — JOIN (Recommended)
SELECT s.student_name
FROM   STUDENTS s
JOIN   ENROLLMENTS e ON s.student_id = e.student_id
WHERE  e.course = 'Maths';
Method 2 — Old-style WHERE (Implicit JOIN)
SELECT s.student_name
FROM   STUDENTS s, ENROLLMENTS e
WHERE  s.student_id = e.student_id
AND    e.course = 'Maths';
Method 3 — Subquery
SELECT student_name
FROM   STUDENTS
WHERE  student_id IN (
    SELECT student_id FROM ENROLLMENTS WHERE course = 'Maths'
);
ApproachReadabilityPerformanceWhen to Use
JOINBestBestMost cases — preferred modern style
WHERE (implicit)Old-styleSameAvoid — hard to read, easy to miss conditions
SubqueryMediumSlowerWhen JOIN would be complex, or for EXISTS checks

Joining Multiple Tables — 3+ Tables

🌍 Real Life Example

A hospital has: PATIENTS → APPOINTMENTS → DOCTORS. To show "Which patient met which doctor on which date" — you need to JOIN all 3 tables together.

School Example — Students + Enrollments + Teachers

Adding a TEACHERS table: each enrollment has a teacher assigned.

teacher_idteacher_namesubject
T1Mr. SharmaMaths
T2Ms. GuptaScience
T3Mr. KhanEnglish
 
-- Join 3 tables: Students + Enrollments + Teachers
SELECT
  s.student_name,
  e.course,
  e.marks,
  t.teacher_name
FROM     STUDENTS s
JOIN     ENROLLMENTS e  ON s.student_id = e.student_id
JOIN     TEACHERS t     ON e.teacher_id = t.teacher_id;
RuleFor each additional table, add another JOIN...ON block. Chain them one by one. Order generally doesn't affect result for INNER JOIN, but matters for LEFT/RIGHT JOIN.
💡 Formula for N tables
You need N-1 JOINs to join N tables. 3 tables → 2 JOINs. 5 tables → 4 JOINs.

Common Mistakes & Exam Tips

❌ Common Mistakes
  • Forgetting ON clause — gives accidental CROSS JOIN (millions of rows!)
  • Ambiguous column names — if both tables have id, write s.id not just id
  • NATURAL JOIN with wrong columns — if both tables share a "name" column, it joins on that too!
  • Expecting CROSS JOIN to be small — 1000 × 1000 = 1 million rows
  • MySQL FULL OUTER JOIN — MySQL doesn't support it, use UNION trick
✅ Exam Tips
  • INNER JOIN = intersection (middle of Venn)
  • LEFT JOIN always includes ALL rows of left table
  • NULL in result means no match was found
  • CROSS JOIN has no ON clause — Cartesian product
  • SELF JOIN requires alias for the table
  • N tables → N-1 JOINs needed
  • EQUI JOIN uses =, NON-EQUI uses <, >, BETWEEN
  • LEFT JOIN + WHERE IS NULL = rows in Left NOT in Right
Important Distinctions — Frequently Asked in Exams
QuestionAnswer
INNER JOIN vs EQUI JOININNER JOIN is a type of JOIN. EQUI JOIN describes the condition (using =). An INNER JOIN with = is an EQUI JOIN.
LEFT JOIN vs LEFT OUTER JOINSame thing! OUTER is optional keyword.
NATURAL JOIN vs INNER JOINNATURAL JOIN auto-detects common columns. INNER JOIN you specify ON condition manually.
Can CROSS JOIN have WHERE?Yes! WHERE filters the Cartesian product. CROSS JOIN + WHERE = similar to INNER JOIN.
Which JOIN returns most rows?CROSS JOIN (m×n rows), then FULL OUTER JOIN, then LEFT/RIGHT JOIN, then INNER JOIN (fewest).

Quick Reference Summary — All JOINs at a Glance

🔵INNER JOINOnly rows that match in BOTH tables. Most common. Like intersection.
🟢LEFT JOINAll rows from LEFT + matched from RIGHT. Unmatched right = NULL. Left table is boss.
🟠RIGHT JOINAll rows from RIGHT + matched from LEFT. Opposite of LEFT JOIN.
🔴FULL OUTER JOINALL rows from BOTH tables. Unmatched = NULL on the other side.
🔵CROSS JOINEvery row × every row. No ON clause. m × n rows result. Cartesian product.
🟡SELF JOINTable joined with itself using two aliases. Used for hierarchical data (manager-employee).
🟣NATURAL JOINAuto-detects common column names. No ON clause. Risky if tables share multiple column names.
🟢EQUI JOINAny JOIN using = in ON condition. Not a keyword — describes condition type.
🟠NON-EQUI JOINJOIN using <, >, BETWEEN in ON condition. Used for range-based matching.
