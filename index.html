SQL Command Categories — DDL, DML, VDL

Core ConceptSQL commands are divided into 3 categories based on what they operate on.
DDL — Data Definition Language

Operates on the structure of the database (schema).

  • CREATE TABLE — banayi table
  • ALTER TABLE — modify ki table
ShortcutStructure change = DDL
DML — Data Manipulation Language

Operates on the data inside tables.

  • SELECT — data dekhna
  • INSERT — data daalna
  • DELETE — data hataana
ShortcutData ke andar ghusna = DML
VDL — View Definition Language

Used to create views (virtual tables).

  • CREATE VIEW — virtual table banayi
Exam NoteSELECT is technically DQL, but in exams → count it under DML.
CommandCategoryWhy
CREATE TABLEDDLCreates table structure
SELECTDMLRetrieves data
INSERTDMLAdds data to table
CREATE VIEWVDLCreates virtual table
DELETEDMLRemoves data
ALTER TABLEDDLModifies table structure

NATURAL JOIN vs Normal JOIN — deptID Trap

The Big DifferenceNATURAL JOIN removes duplicate columns. Normal WHERE join keeps both copies of the common column.
NATURAL JOIN
SELECT * FROM Employee E
NATURAL JOIN Department D;

Detects common column (deptID) automatically. Removes duplicate.

Output Columns
empID, deptID (once), empSalary, empHobby, deptName, deptFloor
WHERE JOIN (Old Style)
SELECT * FROM Employee E,
Department D
WHERE E.deptID = D.deptID;

Manual condition. Does NOT remove duplicates.

Output Columns
empID, E.deptID, empSalary, empHobby, D.deptID, deptName, deptFloor
Query TypedeptID appears
NATURAL JOIN1 time (duplicate removed)
WHERE JOIN2 times (both copies kept)

Foreign Keys — Complete Guide

DefinitionA Foreign Key (FK) is a column that references the Primary Key of another table (or the same table). It maintains Referential Integrity.
Real-Life Analogy
  • Aadhaar number = PK (unique identity)
  • Bank account stores Aadhaar = FK (must exist in Aadhaar DB)
  • If Aadhaar doesn't exist → account invalid
SALESPERSON Database — FK Example
TableFK ColumnReferencesWhy
SALESPERSONsupervisorSSNSALESPERSON(SSN)Supervisor is also an employee — self-reference
TRIPSSNSALESPERSON(SSN)Trip belongs to a salesperson
EXPENSEtripIDTRIP(tripID)Expense belongs to a trip
EXPENSEaccountNoACCOUNT(accountNo)Assumption: ACCOUNT table exists
Key InsightEXPENSE is a bridge table handling Many-to-Many between TRIP and ACCOUNT. Without it, one trip could only link to one account.
FK RulesFK value must either EXIST in the referenced PK column, or be NULL. Random values → Referential Integrity violation.

Q(i): Suppliers of RED parts
-- RA Expression
π sName (
  Suppliers ⨝ (

Functional Dependency — Core Concept

DefinitionA → B means: if two rows have the same A value, they MUST have the same B value. A "determines" B.
Real-Life Example
RollNoName
101Ram
102Shyam
101Aman ❌

RollNo → Name (same ID → same name always). Row 3 VIOLATES this FD.

Types of Dependency
1
Full Dependency — depends on entire key. Example: (A,B) → C
2
Partial Dependency — only part of key. Example: A → D (when key is A,B)
3
Transitive Dependency — chain. Example: A→D, D→E (so A→E indirectly)
Quick Violation Check Method
1
Does the LHS value repeat in any two rows?
2
If YES → check if RHS is also same in those rows
3
If RHS differs → FD VIOLATED. If same → OK.

Closure & Partial Dependency — PYQ Solved

Given: R = {A,B,C,D,E,F,G,H,I,J}, FDs:
AB → C,  A → D,E,  B → F,  F → G,H,  D → I,J
Closure of {A,B} — Step by Step
1
Start: {A, B}
2
Apply AB → C → Add C → {A, B, C}
3
Apply A → D, E → Add D, E → {A, B, C, D, E}
4
Apply B → F → Add F → {A, B, C, D, E, F}
5
Apply F → G, H → Add G, H → {A, B, C, D, E, F, G, H}
6
Apply D → I, J → Add I, J → {A, B, C, D, E, F, G, H, I, J}
Result(AB)⁺ = {A, B, C, D, E, F, G, H, I, J} → AB determines all attributes → AB is the candidate key.
Partial Dependency Check

Primary Key = (A, B). Check if any non-prime attribute depends on PART of key:

FDLHS is...?Verdict
A → D, EOnly A (part of key)❌ PARTIAL DEPENDENCY
B → FOnly B (part of key)❌ PARTIAL DEPENDENCY
AB → CFull key✔ Full Dependency
ConclusionYES — relation has partial dependency → VIOLATES 2NF.

BCNF — Boyce-Codd Normal Form

BCNF RuleFor every FD X → Y, X must be a SUPER KEY. If any FD's LHS is NOT a super key → BCNF violated.
R(A, B) — Is it in BCNF?
CaseFDLHS is Super Key?BCNF?
Case 1No FDN/A✔ Yes
Case 2A → BYes (A is key)✔ Yes
Case 3B → AYes (B is key)✔ Yes
ConclusionA 2-attribute relation is ALWAYS in BCNF. BCNF violations start with 3+ attributes.

Same info in multiple rows. Update one → forget others → inconsistency.

Normalization — 1NF → 2NF → 3NF

Normal FormRuleWhat to Remove
1NFAll cells must have atomic (single) valuesMulti-values, repeating groups
2NFMust be in 1NF + No partial dependencyDependency on part of composite key
3NFMust be in 2NF + No transitive dependencyA→B→C chains (indirect dependency)
CANDIDATE Example — Full 1NF → 2NF → 3NF
── ORIGINAL (UNNORMALIZED) ──
CANDIDATE (Ssn, Name, {JOB_HIST (Company, High_pos)})
Problem{ } = repeating group → violates 1NF
── STEP 1: 1NF ──

Remove repeating groups. Create one row per job:

SsnNameCompanyHigh_pos
1RamTCSDev
1RamInfosysLead
2ShyamWiproManager
Remaining ProblemName repeats for same Ssn → partial dependency (Ssn → Name, but key is (Ssn, Company))
── STEP 2: 2NF ──

Split tables to remove partial dependency:

CANDIDATE (Ssn, Name)

SsnName
1Ram
2Shyam

JOB_HISTORY (Ssn, Company, High_pos)

SsnCompanyHigh_pos
1TCSDev
1InfosysLead
── STEP 3: 3NF ──

If Company → High_pos (same company always gives same role) → transitive dependency exists. Fix:

CANDIDATE (Ssn, Name)

COMPANY (Company, High_pos)

JOB_HISTORY (Ssn, Company) → link table

Final DesignCANDIDATE (Ssn, Name) + COMPANY (Company, High_pos) + JOB_HISTORY (Ssn, Company)
1NF Trap — Convert CANDIDATE Relation (PYQ)

Given: CANDIDATE (Ssn, Name, {JOB_HIST (Company, High_pos)})

Convert to 1NF: Remove { } by creating separate rows:

CANDIDATE (Ssn, Name, Company, High_pos)
-- Each row = one job of one candidate
English Exam AnswerThe relation contains a repeating group {JOB_HIST} which violates 1NF. To convert, remove the repeating group and create separate tuples for each job history. Result: CANDIDATE(Ssn, Name, Company, High_pos).

Data Independence & 3-Schema Architecture

Core ConceptAbility to modify the database schema at one level WITHOUT affecting the next higher level.
Real-Life Analogy

App stored data in Excel → migrated to MySQL → App still works, user noticed nothing. That's Data Independence.

Physical Data Independence

Change internal storage structure without affecting conceptual schema.

Example: Hard disk → SSD, adding indexes, changing file format. Queries remain same.

Logical Data Independence

Change conceptual schema without affecting external schema / app.

Example: Adding Email column to Student table. Existing app still runs same queries.

NoteLogical independence is HARDER to achieve than physical.
3-Schema Architecture + Schema Definition Languages
LevelNameWhat it definesLanguage
ExternalUser ViewWhat each user sees (views)VDL (View Definition Language)
ConceptualLogical SchemaAll tables, relationships, constraintsDDL (Data Definition Language)
InternalPhysical SchemaHow data is actually stored (indexes, files)SDL (Storage Definition Language)
RealityIn modern DBMS: you mostly use DDL. VDL (views) occasionally. SDL is mostly hidden/automatic.

Concurrency — Lost Update Problem

Lost Update ProblemTwo transactions read same data → update independently → one update overwrites the other → one update is LOST.
Bank Account Example — X = 5 (PYQ)
TimeT1T2X in DB
1read(X) → 55
2X = 5+10 = 15 (local)5
3write(X) = ???unclear
4read(X) → 5 (old value!)5
5X = 5+20 = 255
6write(X) = 2525
7commit25
8commit25
ProblemT1's update (+10) is LOST. Final = 25 instead of correct 35. This is the Lost Update Problem.
How to Fix — Prevention Techniques
MethodHow it prevents
LockingOne transaction holds lock → others wait → no conflict
2PL (Two-Phase Locking)Acquire all locks first, release after commit
Serializable IsolationTransactions behave as if run one after another
Timestamp OrderingOlder transaction gets priority, conflict → rollback
Optimistic ConcurrencyAssume no conflict → check at end → rollback if conflict
PYQ — Schedule Problem (T1, T2, X=5)

Problem identified: Dirty Read + Lost Update

T2 reads old value of X (5) before T1 commits. T2 computes 5+20=25 and writes. T1's addition of 10 is completely lost.

Final Value of X = 25 (incorrect — should be 35)

File System vs Database — When to Use What

FeatureFile SystemDatabase (DBMS)
Data typeUnstructured (files, images)Structured (tables, relations)
ConcurrencyWeakStrong (locks, transactions)
Query supportBasicPowerful (SQL)
Data integrityWeakStrong (constraints)
Multi-user accessProblematicHandled well
Large binary filesFast & easyOverhead
Employee Management System → DBMS

Multiple users (HR, Manager), complex queries (salary > 50k), consistency needed, relationships exist → Database is better.

CAD Tools → File System

Large binary design files, mostly single-user, no complex SQL queries needed, fast file access required → File System is better.

Decision RuleStructured + multi-user + complex queries → DBMS. Large files + single-user + no complex queries → File System.

ON UPDATE CASCADE / ON DELETE SET NULL

Tables: PART (parent) → SUPPLIER (child via Pnumber FK)
ON UPDATE CASCADE

Parent PK changes → child FK automatically updates.

-- UPDATE PART SET Pno='P10'
--   WHERE Pno='P1'

-- SUPPLIER auto-updates:
Pnumber: P1 → P10 ✔

Think: "Parent changed → child syncs."

ON DELETE SET NULL

Parent row deleted → child FK becomes NULL.

-- DELETE FROM PART
--   WHERE Pno='P1'

-- SUPPLIER:
Pnumber: P1 → NULL

Think: "Parent gone → child's link becomes empty."

ConditionOnly works if FK column allows NULL.
OptionOn parent UPDATEOn parent DELETE
CASCADEChild FK auto-updatesChild row also deleted
SET NULLChild FK → NULLChild FK → NULL
RESTRICTUPDATE blockedDELETE blocked
NO ACTIONError thrownError thrown
